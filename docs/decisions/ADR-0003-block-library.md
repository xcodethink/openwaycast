# ADR-0003 · 模板数据化走"区块库"(Phase 1.2 架构)

- 状态:已接受
- 日期:2026-06-17

## 背景(关键发现)
对比两家品牌的 8 镜 compositions 骨架:
- **封面(01)、封底(08)两家同构**;
- **镜 02–07 两家不同构** —— 因为它们把同样的文件槽位映射成了**不同的版式**(例:WJ 镜3=三层堆叠图,HT 镜3=社论式;WJ 镜2=人物速写,HT 镜2=痛点清单)。

所以两家**不是"同一模板换文案+换色"**,而是**用了不同的版式区块**。

## 决策
Phase 1.2 走 **区块库(block library)**,而不是"逐品牌就地抽文案":
- 抽出一套可复用**版式区块**(已在两家中存在的:`cover` / `editorial-statement+list` / `who-editorial` / `layer-stack` / `tile-grid` / `step-flow` / `manifesto` / `cta` …)。
- 每个区块 = 一个**数据驱动模板**(内容来自 JSON 槽位 + 品牌色经 CSS 变量注入)。
- 一个品牌 = 一个 **storyboard**:有序的 `[{block, content}]` 列表 + `brand.json`(色)+ 背景图。
- WJ / HT 改写成两个 storyboard 配置以做对等验证。

## 理由(为什么不是逐品牌模板化)
产品目标是"**任何公司 URL → 自动出片**"。若每家公司都要**手写 8 个 composition**,自动化(Phase 2)就无从谈起——LLM 不擅长生成可靠的版式 HTML。**必须有固定的可复用区块,自动化只负责"选区块 + 填内容"**。因此区块库不是可选项,是产品的地基。逐品牌模板化是半条死路(迁到区块库时大部分要返工)。

## 影响
- Phase 1.2 工作量上调(从"抽文案"变成"建区块库 + storyboard schema + 改写两家"),预估 2–3 个对话。
- Phase 2 内容自动化随之清晰:`brief → 选区块序列 + 填每块内容 + 选图配色`。
- 现有两家成片是**对等验证基准**:区块库版必须逐帧复现。

## 待定(实施时定)
- 区块 schema 字段命名;storyboard 文件格式(JSON/YAML)。
- 区块内的入场动画/节奏是否参数化(默认沿用现有,先不参数化)。
